3. ¿QUÉ MÁS SE PUEDE MEJORAR?
-------------------------------

A. PRE-FILTER DE CUENTAS MUERTAS (ALTA PRIORIDAD)
   ------------------------------------------------
   - 60390992, 62615541 (JOSHEP), 73968815 (AARON): fallan TODOS los días
   - Propuesta: archivo `cuentas_expiradas.json` que se actualiza automáticamente
   - Si una cuenta falla 3 días seguidos con código 404 → skip permanente
   - Ahorro: ~18 requests/día que van a cuentas muertas
   - Implementación: script aparte que revisa logs históricos y genera lista negra

B. BURST ADAPTATIVO (MEDIA PRIORIDAD)
   ------------------------------------
   - Hoy: esperas fijas de 2s entre burst
   - Propuesta: medir en tiempo real el ratio de HTTP 500
   - Si las 500 caen a 0 antes de 2s → disparar Burst 2 inmediatamente
   - Si todavía hay muchas 500 después de 2s → extender espera a 3s
   - Implementación: Promise.race entre timeout fijo y detección de "pausa de 500"
   - Beneficio potencial: ganar ~0.5-1s dentro de la ventana

C. MULTI-SHOT PARA TODAS LAS CUENTAS (BAJA PRIORIDAD)
   ----------------------------------------------------
   - Hoy: VIPs tienen 3 shots/burst, normales 1
   - Si mañana el éxito es < 90%: probar 2 shots para todas
   - Riesgo: duplica tráfico (340 requests en vez de 170)
   - Beneficio: ~5-10% más de probabilidad si el 500 es aleatorio
   - Solo si es necesario después de ver resultado de mañana

D. SOCKET PRE-WARMING (MEDIA PRIORIDAD)
   ---------------------------------------
   - El warmup actual abre 1 socket (cuenta sonda)
   - Propuesta: durante warmup, hacer 3-5 requests paralelas para pre-crear
     sockets TCP en el pool de Keep-Alive
   - Así Burst 1 no tiene que esperar que se abran sockets nuevos
   - Ya implementado implícitamente: el warmup abre socket, Burst 1 lo reusa
   - Mejora: worker-side warmup con múltiples sockets antes de recibir orden

E. COMPRESS BURST TIMING (BAJA PRIORIDAD)
   ----------------------------------------
   - RTT: ~650ms. Si Burst 1 falla rápido en 1s, podríamos:
     - Burst 1: t=0
     - Burst 2: t=1000 (en vez de 2000)
     - Burst 3: t=2500 (en vez de 4000)
   - Esto deja margen si la ventana se cierra antes de 5s
   - Riesgo: si MySQL deadlock tarda 2s en liberarse, Burst 2 muy temprano = waste
   - Solo recomendable si mañana el burst 2 no recupera nada en 2000ms

F. DISTRIBUCIÓN POR LATENCIA (BAJA PRIORIDAD)
   --------------------------------------------
   - Hoy: distribución equitativa (4 cuentas/worker)
   - Propuesta: workers rápidos (worker-8: 13s) reciben más cuentas
   - Workers lentos (worker-9: 390s) reciben menos
   - Beneficio: reduce el arrastre total
   - Complejidad: requiere medir latencia en cada ronda

G. ELIMINAR VISUAL POR COMPLETO (EVALUAR MAÑANA)
   ------------------------------------------------
   - Si RAW ≥ 92%: visual es redundante
   - Si RAW = 95%+ y el visual sigue ejecutándose: ahorra ~18 min de ejecución
   - Decisión: esperar resultado de mañana

H. WARMUP SIN LOGIN (MUY BAJA PRIORIDAD)
   ----------------------------------------
   - Si Keep-Alive funciona, el primer request del warmup abre el socket
   - Pero si la API no requiere cookie/sesión (Express), ni siquiera
     necesitamos warmup con login — solo un HEAD o GET a cualquier endpoint
     para abrir el socket TCP
   - Ahorro: ~1 request de warmup por worker
